# PlanTrip Travel Planner

여행 정보 탐색부터 장소 확인, 일정 구성, 저장·관리까지 이어지는 **Spring Boot 기반 웹 여행 플래너 프로젝트**입니다.

이 프로젝트는 KDT 부트캠프 4인 팀 프로젝트로 진행되었으며, 저는 여행 정보 데이터 수집·정리, 서비스 화면과 DB 구조에 맞는 데이터 분류, 요구사항 분석서·화면설계서·DB 설계 자료 기반의 기능 흐름 정리에 참여했습니다.

---

## 1. 프로젝트 소개

**PlanTrip**은 여행을 준비하는 사용자가 관광지, 맛집, 숙소, 문화·액티비티 정보를 탐색하고, 지도 기반 위치 정보를 참고하여 직접 여행 일정을 구성하고 저장·관리할 수 있도록 기획된 웹 기반 여행 플래너 서비스입니다.

단순히 여행지 정보를 보여주는 서비스가 아니라,
**여행 정보 탐색 → 장소 확인 → 일정 구성 → 저장 및 관리**로 이어지는 흐름을 중심으로 설계되었습니다.

사용자는 장소 정보를 카테고리별로 확인하고, 지도에서 위치를 확인한 뒤, 관심 있는 장소를 일정에 담아 자신만의 여행 계획을 구성할 수 있습니다.

※ 이 README는 전체 기능을 제가 모두 직접 개발했다는 의미가 아니라, 제가 맡은 **데이터 수집·정리, 문서화, DB 구조 이해, 서비스 흐름 정리 역할**을 중심으로 작성했습니다.

## 3. 내가 맡은 역할

이 프로젝트에서 제가 맡은 핵심 역할은 **데이터 수집과 정리, 문서화 참여, 서비스 흐름 이해 및 구조 정리**입니다.

| 역할            | 수행 내용                                                                                   |
| ------------- | --------------------------------------------------------------------------------------- |
| 데이터 수집·정리     | 관광지, 맛집, 숙소, 기념품, 문화·액티비티 정보를 조사하고 정리                                                   |
| 데이터 항목 분류     | 장소명, 주소, 좌표, 운영시간, 휴무일, 평점, 특징, 출처 등 화면과 DB에서 활용될 항목 정리                                 |
| 문서화 참여        | 요구사항 분석서, 화면설계서, 회의록, 프로그램 정의서 내용을 바탕으로 기능 흐름 정리                                        |
| 사용자 유형별 흐름 이해 | 비회원, 회원, 관리자별로 접근 가능한 기능 차이를 구분                                                         |
| DB 구조 이해      | 회원, 도시, 장소, 일정, 상세 일정, 찜, 후기 데이터가 연결되는 구조 파악                                            |
| 코드 흐름 분석      | Spot, City, SpotStats, TravelPlan, PlanDetail, WishList, Review/Community 구조와 데이터 흐름 이해 |

### 역할 정리

제가 정리한 장소 데이터는 단순한 자료 목록이 아니라,
장소 조회 화면, 지도 위치 표시, 일정 구성, 찜, 후기 기능의 기반 데이터로 연결됩니다.

또한 Spring Boot 기반 코드 구조를 확인하며 Controller, Service, Repository, Entity가 어떤 방식으로 장소 데이터와 일정 기능을 연결하는지 이해했습니다.

## 프로젝트를 통해 배운 점

PlanTrip 프로젝트를 통해 서비스는 화면만으로 완성되는 것이 아니라, 요구사항, 화면 흐름, DB 구조, 데이터 정리가 함께 맞물려야 한다는 점을 배웠습니다.

특히 여행 정보가 화면에 표시되기 위해서는 장소명, 주소, 좌표, 운영시간, 휴무일, 평점 같은 데이터가 정리되어야 하고, 일정 생성이나 찜, 후기 기능이 동작하기 위해서는 회원, 장소, 일정, 상세 일정, 찜, 후기 데이터가 구조적으로 연결되어야 한다는 점을 이해했습니다.

또한 협업 과정에서 문서 확인, 피드백 확인, 역할 조율의 중요성을 느꼈고, 맡은 일을 한 번 더 점검하고 필요한 내용을 제때 공유하는 태도가 중요하다는 점을 배웠습니다.

앞으로는 단순히 화면이나 기능만 보는 것이 아니라, 그 기능을 가능하게 하는 데이터 구조와 사용자 흐름까지 함께 이해하는 개발자로 성장하고자 합니다.
